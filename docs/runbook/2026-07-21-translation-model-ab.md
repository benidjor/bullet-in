# 번역 모델 교체 평가 런북 — A/B 절차 · 채점 축 · 비용 계산 (2026-07-21)

번역에 쓰는 Gemini 모델을 바꿀지 판단할 때 쓰는 절차.
새 모델이 나올 때마다 반복되므로 절차와 함정을 남긴다.
2026-07-21 `gemini-2.5-flash-lite` → `gemini-3.1-flash-lite` 검토에서 실행한 것을 그대로 정리했다.

핵심 아이디어는 **평가 도구를 새로 만들지 않는 것** 이다.
이 프로젝트에는 이미 번역 품질을 판정하는 결정적 검출기가 여러 개 있고 (게이트 4축 · 말투 검출기 · 표기 사전), 그것들을 그대로 채점기로 돌리면 사람 판단 없이 대부분의 축이 수치로 나온다.

## 1. 선행 조건

- **회차 시각을 피한다** — 스케줄 회차 (KST 09 · 15 · 21 · 03) 와 API 키 · 속도 한도를 공유한다.
회차 직후가 가장 안전하다.
- **요금 등급을 먼저 확인한다** — AI Studio 프로젝트 화면에서 등급과 월 지출을 본다.
무료로 착각하면 백필 비용을 실제보다 크게 잡아 결론이 뒤집힌다 (`CLAUDE.md` 함정 절 참조).
- **표본은 VM 에서 뜬다** — 로컬 mart 는 낡아 있다 (2026-07-21 실측 로컬 205행 vs VM 227행 · 41시간 차).

## 2. 표본 선정

- 소스군을 고르게 덮는다 — 라운드업 (bbc_gossip) · 일반 언론사 · 커뮤니티 (fmkorea) · 트윗.
- 같은 소스 안에서는 본문 길이 분포를 고르게 (짧은 · 중간 · 긴).
- 라운드업을 반드시 넣는다 — 단신 나열 구조라 누락 · 압축이 가장 잘 드러난다.
- 12~15건이면 축별 경향이 보인다. 파싱 실패처럼 확률적인 축은 회차를 두 번 돌려 편차를 본다.

## 3. 호출 조건 — 운영 경로와 동일하게

- 프롬프트는 `enrich.TRANSLATE_PROMPT` · `PARAPHRASE_PROMPT` 를 그대로 쓴다.
- `max_output_tokens` · `response_mime_type` 도 운영값과 같게 둔다.
- mode 분기 (`partition_by_paywall`) 도 운영과 같은 규칙으로 나눈다.
- 같은 입력을 두 모델에 각각 보내 **쌍으로** 비교한다.
소스가 다르면 난이도가 달라 모델 차이와 구분되지 않는다.
- 호출 간격을 두어 속도 한도를 넘지 않게 한다.

## 4. 채점 축

기존 검출기를 재사용하는 축이 앞쪽이다.

| 축 | 측정 방법 | 해석 |
|---|---|---|
| 파싱 성공률 | `enrich._extract_full` 반환이 `None` 인 비율 | 실패 = 그 회차 스킵 → 다음 회차로 지연. 비용에도 직접 영향 |
| 게이트 플래그 | `detect_title_hallucination` · `detect_title_mistranslation` · `detect_roundup_omission` · `detect_club_injection` | 환각 · 누락의 결정적 신호 |
| 말투 위반 | `tone.has_polite_ending` | 요약 2필드는 말투 백필이 잡지만 **본문은 아무도 안 잡는다** — 본문도 따로 셀 것 |
| 표기 교정량 | `glossary.yaml` 키가 산출물에 등장한 횟수 | 낮을수록 통용 표기를 스스로 맞춘 것 |
| 본문 완전성 | `body_ko` 자수 / 원문 자수 | 참고용 — §6 함정 참조 |
| 사실 보존 | 원문의 이적료 · 나이 숫자가 번역에 남은 비율 | 자수비보다 신뢰할 수 있는 완전성 신호 |
| 지연 · 토큰 | 호출 시간 · `usage_metadata` | 비용 계산 입력 |
| 원문 제목 폴백 | `title_ko = title_original` 행 수 | **교체 전후 비교 필수** — 늘었으면 게이트 오탐을 의심한다 (§6) |

## 5. 블라인드 판정 (사람 · LLM 심사관)

수치로 안 갈리는 축 (오역 · 창작 · 가독성) 은 눈으로 봐야 한다.

- 원문 + 번역 A + 번역 B 를 한 파일에 담되 **A · B 배정을 무작위로** 하고 키를 따로 저장한다.
해시 패리티 같은 재현 가능한 규칙을 쓰면 나중에 검증할 수 있다.
- 판정 기준을 미리 준다 — 사실 완전성 · 정확성 · 문체 · 표기 · 가독성.
- **홍보 문구 · 내비게이션 · 이미지 캡션은 빼는 것이 정답** 이라고 명시한다.
안 그러면 심사관이 그것까지 "누락" 으로 센다.
- 판정 후 키를 붙여 해독한다.

## 6. 함정

- **자수비는 홍보 잡문에 오염된다** — `body_source` 에는 구독 유도 · 팟캐스트 안내 같은 기사 무관 문구가 섞여 있다.
그것을 걷어낸 모델은 자수비가 낮게 나오는데 이는 오히려 정답이다.
자수비 단독으로 완전성을 판정하지 말고 수치 보존율 · 블라인드 판정과 함께 볼 것.
- **표기 교정량은 본문 길이에 비례한다** — 행당으로 세면 본문이 긴 쪽이 불리하다.
1,000자당으로 정규화해 비교한다.
- **압축은 모델 결함이 아닐 수 있다** — 요약하듯 줄여 쓰는 모델은 프롬프트에 완역 지시 한 줄을 더해 재시험한다.
2026-07 검토에서 이 한 줄로 자수비 0.17~0.27 이 0.37~0.44 로 회복됐다.
프롬프트로 고쳐지는 것을 모델 탓으로 돌리면 잘못된 결론이 난다.
- **모델을 바꿔도 기존 행은 재번역되지 않는다** — 대상 선정이 `WHERE title_ko IS NULL` 이라 새로 수집된 것만 새 모델로 번역된다.
표기가 두 모델로 섞이는 것을 감수하거나 백필해야 한다 (§8).
- **새 모델은 게이트를 새로 자극한다** — 품질이 좋아져도 문체가 달라지면 기존 검출기가 오탐하기 시작한다.
2026-07 교체에서는 제목이 간결해지자 인명 누락 검사가 7건을 잘못 잡아 영문 제목이 화면에 노출됐다.
테스트도 로그도 정상이라 폴백 행 수를 세지 않으면 모른다.
진단과 해법은 `docs/troubleshooting/2026-07-22-model-swap-gate-false-positives.md`.
- **서빙에 런타임 결합이 있다** — 가십 단신 카드는 번역문 문단 끝의 `(출처)` 표기를 원문 표지 집합과 대조해 만든다.
모델이 그 표기를 번역하거나 위치를 옮기면 **git 충돌도 테스트 실패도 없이** 항목화만 사라진다.
교체 전후로 `재번역 큐 … 단신누락=` · `라운드업 단신 누락 잔존` 로그 건수를 비교하면 같은 신호를 잡을 수 있다.
다만 원문에 표지가 없는 기사 (일정표 등) 는 두 축 모두 0이라 이 방법으로는 회귀가 안 잡힌다.

## 7. 비용 계산

- 콜당 입력 · 출력 토큰은 `usage_metadata` 실측값을 쓴다.
- **실효 콜 수 = 1 / (1 − 파싱 실패율)** — 실패한 행은 다음 회차에 다시 호출되므로 단가가 낮아도 총액이 커진다.
- 월 비용 = 일 신규 기사 수 × 30 × 실효 콜 × (입력 토큰 × 입력 단가 + 출력 토큰 × 출력 단가).
- 계산 결과를 AI Studio 의 실제 월 지출과 대조해 검산한다.
2026-07 검토에서 추정 ₩700 · 실측 ₩676.74 로 맞았다.

## 8. 교체 후 백필 (선택)

표기를 한 모델로 통일하려면 기존 행을 다시 번역해야 한다.

- **스냅샷을 먼저 뜬다** — VM 의 MariaDB 에서 `content_hash` · `source_id` · `title_original` · 번역 4필드.
되돌리려면 옛 모델이 그때도 살아 있어야 하므로 재실행으로 복구된다고 보면 안 된다.
- **번역 4필드를 모두 NULL 로 민다** — `title_ko` 만 밀면 `summary_ko` 가 남아 `run.py` 의 `retry` 판정이 참이 된다.
그러면 모든 행이 재시도 행으로 취급돼 게이트의 1차 검출 (재번역 큐) 을 건너뛰고, 오검출이 곧바로 원문 제목 폴백으로 확정된다.
- 실행은 하루 4회 스케줄을 기다릴 필요 없이 enrich 전용 패스로 돌린다 — 절차 전문은 `docs/runbook/2026-07-19-enrich-only-pass.md` §5 (스냅샷 · 4필드 NULL · 실행 · 롤백).
- `transfer_stage` 는 대상이 아니라 재분류 비용이 들지 않는다.

### 8.1. 백필 후 표기 대조

백필하면 표기가 한 모델로 통일되지만, 그 통일된 표기가 통용 표기라는 보장은 없다.
어떤 표기가 바뀌었는지 뽑아 사전에 올릴 것을 고른다.

- **본문을 축으로 삼는다** — 제목은 새 모델이 통째로 다시 써서 토큰 차이가 표기 변화와 구분되지 않는다.
같은 기사의 옛 번역과 새 번역을 `content_hash` 로 이어 본문끼리 대조한다.
옛 번역은 스냅샷 덤프를 임시 테이블로 되살려 쓴다 (enrich 런북 §5.4 와 같은 방식).
- **표기 변형은 글자가 치환된 쌍만 인정한다** — 뒤에 뭔가 붙고 떨어진 쌍 (`촐리스` · `촐리스와`) 은 조사 · 파생이라 걸러야 한다.
이 한 줄로 노이즈가 절반 이하로 줄었다 (2026-07 실측 202쌍 → 126쌍).
- **세 갈래로 나눈다** — 사전 동의 그룹으로 짝지어지는 확정 쌍 · 사전 밖 신규 변형 · 짝을 못 지은 것.
사전이 흡수하는 확정 쌍은 손댈 필요가 없고, **목록을 깨는 것은 사전 밖 신규 변형**이다.
- **한계를 함께 적는다** — 공백이 든 구단명 (`클뤼프 브뤼허`) 은 토큰이 갈라져 안 잡힌다.
짝을 못 지은 목록은 새 모델이 문장을 다시 쓰며 생긴 일반 어휘 교체가 대부분이라 그대로는 쓸 수 없다.

대조 결과를 그대로 등재하지 말 것.
**대부분은 옛 모델과 새 모델의 차이일 뿐이고, 백필이 끝난 시점의 데이터는 이미 한 표기로 통일돼 있다.**
2026-07 검토에서 126쌍 중 화면에서 실제로 갈리던 것은 5건뿐이었다.
등재 전에 현재 데이터의 분포를 다시 세고, 부분 문자열 착시를 걸러 낸다 (`docs/troubleshooting/2026-07-22-glossary-substring-traps.md`).

## 9. 부록 — 2026-07-21 실측 (2.5 vs 3.1 flash-lite)

동일 입력 21쌍 + 블라인드 판정 6건.

| 지표 | 2.5-flash-lite | 3.1-flash-lite | 3.1 + 완역 지시 |
|---|---|---|---|
| 파싱 실패 | 7/21 (33%) | 1/21 (5%) | 0/12 |
| 본문 / 원문 자수비 | 0.43 | 0.35 | 0.44 |
| 원문 수치 보존율 | 69% | 53% | 72% |
| 본문 존댓말 | 12줄 | 0 | 0 |
| 표기 교정 (1,000자당) | 1.49 | 0.76 | 0.91 |
| 평균 지연 | 8.8초 | 3.9초 | — |

- 2.5 의 파싱 실패는 같은 어미를 반복하다 JSON 이 깨지는 응답 붕괴였다.
같은 기사를 두 번 돌렸을 때 본문 분량이 0.08배와 0.90배로 갈릴 만큼 편차도 컸다.
- 블라인드 판정은 케이스 승패가 3 대 3 이었으나 지는 이유가 달랐다.
2.5 는 정확성에서 6건 중 5건 열세였고 (제목에 없는 선수 창작 · 존재하지 않는 출처 표기 · 실존하지 않는 인명 · 의미 역전), 3.1 의 약점은 압축 하나였다.
그 압축은 프롬프트 한 줄로 해소됐다.
- 비용은 3.1 이 약 2.6배 (월 ₩677 → 약 ₩1,750) 였고 절대액이 작아 결정을 가르는 축이 되지 못했다.
