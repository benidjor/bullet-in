# 전 소스 커버리지 감사 — 라이브 대조 절차 (2026-07-24 실측)

"있어야 하는데 없는 기사" 를 소스별로 정량화하는 읽기 전용 감사 절차다.
내부 신호 (오류 · 볼륨 · 신선도) 로 잡히지 않는 조용한 기아는 외부 기준과의 대조로만 드러난다
— 2026-07-24 감사에서 arsenal 피드 동결 · fmkorea IP 차단이 이 방법으로 확정됐다.

## 1. 원칙

- **읽기 전용** — 코드 · DB 무변경. 산출물은 소스별 표와 스코프 제안.
- **어댑터 로직 재사용** — 셀렉터 · 제목 필터 · `_accept` 를 감사 스크립트에 복제하지 말고 import 해서 쓴다.
규칙을 옮겨 적으면 감사 자체가 드리프트한다 (스니펫 드리프트와 같은 함정).
- **접촉 예산 준수** — fmkorea 는 직전 접촉 후 2시간 대기 · X 는 raw 로 대체 (쿠키 소모 0)
· 나머지는 리스트 페이지 1회 GET 수준.

## 2. 절차

1. **VM DB 인벤토리** — 소스별 행수 · `MIN/MAX(fetched_at)` · precision NULL 수 · 전 행 URL 덤프.
`pipeline_runs` 의 최근 12회 `source_counts` 로 "신규 0" 이 dedup 인지 수집 실패인지 1차 분류.
2. **로그 대조** — 신규 0 소스는 `journalctl` 에서 해당 어댑터의 오류 줄 (430 · 스킵 · 드리프트) 을 기간 grep.
3. **라이브 리스트 대조** — 소스별 리스트 · API 를 어댑터와 같은 필터로 읽어
"필터 통과인데 DB 에 없는" 항목을 뽑는다.
4. **누락 후보 3분류** — 아래 §3. 이걸 건너뛰면 설계 동작을 실누락으로 오판한다.
5. **표본 정밀 감사** — journalist 분포 눈검사 + 의심 표기의 tier 해석 대조,
레거시 precision 행은 URL 재방문 표본으로 백필 수율 실측.

## 3. 누락처럼 보이는 것의 3분류 (실측 함정)

리스트에 있는데 DB 에 없는 항목은 대부분 실누락이 아니다.
2026-07-24 감사에서 football_london 18건 차이가 전부 아래로 해소됐다.

- **설계 드랍** — `journalist_allowlist` · 여자팀 필터 등 파이프라인 규칙의 의도된 제외.
바이라인 · 본문을 확인해야 판별된다 (리스트만으론 불가).
- **회차 이후 발행** — 발행 시각이 마지막 회차 뒤면 다음 회차 대상이지 누락이 아니다.
- **도입 전 창 밖** — 어댑터 도입일 이전 발행 + 현재 창 (page-size · 페이지 수) 밖이면 설계상 미수집.

셋을 제하고 남는 것만 실누락으로 보고한다.

## 4. 판정 보조 신호

- **같은 초 무더기** — `published_at` 이 초 단위로 수십 건 몰리면 레거시 폴백 (수집 시각 저장) 의 화면 왜곡 증거.
- **교차 커버리지** — 실누락 후보의 사건이 타 소스 행에 있는지 검색
→ 있으면 사건이 사라진 게 아니라 그 소스만의 관점이 사라진 것 (스코프 판단이 달라짐).
- **아카이브 유무** — 소급 복원 가능성: API 아카이브 (arsenal · guardian) · 포럼 검색 (fmkorea) 은 가능,
리스트 페이지 (bbc · sky 등) 는 롤오프 후 불가.
