# 표기 사전의 부분 문자열 함정 세 가지 (2026-07-22)

`config/glossary.yaml` 은 번역 결과에 단순 문자열 치환을 순서대로 거는 사전이다.
구조가 단순한 대신, 항목이 늘수록 서로 겹쳐서 생기는 사고가 나온다.
표기 교정 15건을 등재하면서 세 가지를 만났고, 그중 둘은 실제로 사고로 이어질 뻔했다.

세 가지 모두 **테스트도 안 깨지고 예외도 안 나며 화면에서만 이상하게 보인다**는 공통점이 있다.

## 1. 순서 함정 — 짧은 규칙이 앞서면 긴 규칙이 죽는다

사전은 기재 순서대로 치환된다.
`브뤼헤` → `브뤼허` 를 앞에 두면 `클럽 브뤼헤` 가 먼저 `클럽 브뤼허` 로 바뀌어, 뒤따르는 `클럽 브뤼헤` 규칙이 끝내 걸리지 않는다.
결과는 교정이 절반만 된 상태다.

```yaml
# 나쁨 — '클럽 브뤼헤' 규칙이 영영 안 걸린다
브뤼헤: 브뤼허
클럽 브뤼헤: 클뤼프 브뤼허

# 좋음 — 구단명이 붙은 긴 표기를 먼저
클럽 브뤼헤: 클뤼프 브뤼허
브뤼헤: 브뤼허
```

이 함정은 이미 알려져 있었다 (`메리에르` · `메리에` 항목에 주석이 달려 있다).
그럼에도 재발했으므로 주석 대신 테스트로 고정했다.

## 2. 자기 재적용 함정 — 교정형이 다른 규칙의 대상이 된다

교정 결과가 또 다른 규칙에 걸리면 치환이 누적된다.
`라일리` → `오라일리` 를 넣으면 이미 올바른 `오라일리` 가 `오오라일리` 가 된다.

한 번 돌릴 때는 안 보인다.
소급 적용 (enrich 런북 §6) 은 여러 번 돌릴 수 있는 절차라, 돌릴 때마다 글자가 늘어난다.

**판정 방법은 멱등성이다.**
모든 키와 교정형을 담은 문자열에 사전을 두 번 걸어 1회차와 2회차가 같은지 본다.
`tests/test_enrich.py` 가 실제 설정 파일을 읽어 이 조건을 검사한다.

```python
text = " ".join(list(g) + list(g.values()))
once = apply_glossary({"body_ko": text}, g)
twice = apply_glossary(once, g)
assert twice["body_ko"] == once["body_ko"]
```

위험한 규칙을 넣어 테스트가 실제로 실패하는지 확인해 두면 좋다.
통과만 하는 테스트는 아무것도 지키지 않는다.

## 3. 계수 착시 — 짧은 표기는 긴 표기 안에서도 세어진다

표기가 갈렸는지 셀 때 `count()` 를 쓰면 짧은 쪽이 긴 쪽 안에서도 잡힌다.

- `라일리` 1회 · `오라일리` 1회로 나와 표기가 갈린 것처럼 보였다.
실제로는 그 1회가 `오라일리` 안에 있던 것이라 표기가 갈린 게 아니었다.
- `브뤼헤` 47회 중 44회는 `클럽 브뤼헤` 안이었고 단독은 3회였다.
- `레거` 8회는 전부 `슬레거스` 안, `링구스` 1회는 `에어링구스` (항공사) 안이었다.

**교정 항목으로 올리기 전에 문맥을 직접 본다.**
표기가 갈린 것처럼 보이는 게 착시면 규칙을 넣을 이유가 없고, 넣으면 §2 함정을 만든다.

## 4. 등재 절차

1. **분포를 센다** — 어느 표기가 몇 회 · 몇 행에 있는지.
2. **짧은 표기는 문맥을 확인한다** — 긴 표기 안에서 세어진 것이 아닌지 (§3).
3. **긴 표기를 먼저 배치한다** (§1).
4. **테스트를 돌린다** — 멱등성 · 순서 규칙 (§2).
5. **`club_map` · `name_map` 과 맞춘다** — 사전이 바꿔 놓은 표기가 등록 키와 다르면 게이트가 대상을 못 찾는다.
6. **소급 적용한다** — 사전을 늘려도 저장된 번역은 안 바뀐다 (enrich 런북 §6).
