# source_counts 희소 표현의 소비자 함정 — 0건 회차 생략이 추세를 미화 (2026-07-13)

## 배경

`pipeline_runs.source_counts` 는 회차별 소스 수집량 JSON 인데, **그 회차에 신규 기사를 낸 소스만 키를 가진다** (0건 · 전부 중복이면 키 부재) — 희소 표현이다.
SLO-6 판정 계층 ( `volume_anomalies` ) 은 처음부터 `h.get(sid, 0)` 으로 부재 = 0 으로 처리했다.
알림 embed 개편 (PR #37) 에서 표시 계층 ( `build_anomaly_alert` 의 추세 시퀀스 ) 이 이 표현의 새 소비자가 됐다.

## 증상

최초 구현이 부재 회차를 건너뛰어, 이미 죽은 소스가 "오늘 갑자기 급락" 한 것처럼 렌더링됐다.

- 최초 코드: `recent = [h[sid] for h in hist[:5] if sid in h]` — 키가 있는 회차만 수집.
- 실패 시나리오: fmkorea 가 직전 회차에 이미 0건이었다면 hist[:5] = [부재, 14, 14, 14, 14]
  → 알림은 `최근: 14 → 14 → 14 → 14 → (오늘) 0` 으로 표시 — 실제로는 한 회차 전부터 죽어 있었다.
- 시퀀스의 존재 이유가 "급락인지 서서히 죽었는지 구분" 인데, 그 신호를 정확히 왜곡하는 결함.

## 원인 — "키 부재" 의 의미를 소비자마다 재해석

희소 표현에서 부재의 의미 (= 0건) 가 계약으로 명시되지 않아, 소비자별로 갈렸다.

- 판정 계층은 부재 = 0 으로 옳게 처리 ( `h.get(sid, 0)` ).
- 표시 계층은 부재 = 결측으로 취급해 **생략** — 같은 데이터에서 다른 의미.
- 스펙 §5.2 는 "이상 소스가 hist 에 없으면 시퀀스 줄 생략" 만 정의하고 **부분 부재** 를 미기술 — 모호 지점에서 왜곡되는 쪽이 선택됐다.
- 단위 테스트 픽스처가 전 회차에 키가 있는 형태뿐이라 통과했고, 태스크 단위 리뷰도 픽스처 범위 안에서만 검증했다.

## 발견 — 계층 간 의미 대조

whole-branch 최종 리뷰가 판정 계층 ( `h.get(sid, 0)` ) 과 표시 계층 ( `if sid in h` ) 의 부재 처리를 나란히 놓고 대조하면서 드러났다.
태스크 단위 리뷰는 각 계층을 따로 보므로 이런 계층 간 의미 불일치가 사각이다.

## 해결 — 부재 = 0 렌더링 · 생략은 완전 부재만

```python
        if any(a.source_id in h for h in hist):
            recent = [h.get(a.source_id, 0) for h in hist[:5]]
            seq = " → ".join(str(n) for n in reversed(recent))
            lines.append(f"최근: {seq} → (오늘) {a.today}")
```

- 시퀀스는 부재 회차를 0 으로 포함 — 판정 계층과 의미 일치.
- 줄 생략은 전체 hist 에 소스가 아예 없을 때 (신규 소스) 만.
- 회귀 테스트: 직전 회차 부재 픽스처 → `최근: 14 → 14 → 14 → 14 → 0 → (오늘) 0` 단언 ( `test_build_anomaly_alert_sequence_counts_absent_rounds_as_zero` ).
- fix 커밋은 PR #37 에 포함.

## 예방 — 희소 컬렉션의 부재 의미를 계약처럼 다루기

- 희소 표현 (부재 키가 의미를 갖는 dict · JSON) 의 **모든 소비자** 는 부재를 명시 처리한다 ( `get(k, 기본값)` ).
  "있는 것만 순회" ( `if k in h` 필터 ) 는 곧 데이터 왜곡이다.
- 새 소비자를 붙일 때 기존 소비자 (여기서는 판정 계층) 의 부재 처리와 의미를 맞추고, 다르게 할 거면 스펙에 명시한다.
- 테스트 픽스처에 부재 케이스를 반드시 포함한다 — 전 키 존재 픽스처만으로는 이 계열 결함이 절대 안 잡힌다.
