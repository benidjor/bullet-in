# 런북 스니펫 ↔ 파이프라인 로직 드리프트 — 분기 추가가 문서 사본에 반영되지 않는 함정

- 날짜: 2026-07-19 (트랙 ⑤ 최종 whole-branch 리뷰에서 발견 · 머지 전 fix)
- 결함 클래스: 문서 ↔ 코드 정합 (#38 "관측 지시 ↔ 싱크 부재" 와 같은 계열)
- fix 커밋: PR #66 내 `05fdc9c`

## 1. 증상

- run.py 분류 패스에 오피셜 규칙 분리 (`rule_stage` — 공홈 행은 LLM 없이 직접 태깅) 를 추가했는데,
  분류 런북 §2 backfill 스니펫은 여전히 미태깅 행 **전건** 을 `classify_stage_rows` 에 넘기고 있었다.
- 더 위험했던 점: 스니펫 도입 문장은 규칙 분리를 미러하는 것처럼 서술돼 있어, 읽는 사람은 정합하다고 믿게 된다.

## 2. 잠재 피해 (실해 발생 전 차단)

- 공홈 수집이 복구된 상태에서 운영자가 이 스니펫을 그대로 실행하면, LLM 은 official 을 반환할 수 없으므로 (프롬프트에서 제거됨) 공홈 행이 agreed 등으로 오태깅된다.
- `transfer_stage IS NULL` 트리거 특성상 한 번 태깅되면 재분류 전까지 **고착** 된다.
- 불변량 SQL (비공홈 official 검사) 은 방향이 반대라 이 오류 (공홈인데 비-official) 를 잡지 못한다.
- 발견 시점엔 공홈 적재 0건이라 실해가 없었지만, 공홈 복구 시점 = 정확히 이 런북을 따라가는 시점이다.

## 3. 왜 태스크 리뷰를 통과하고 최종 리뷰에서만 잡혔나

- 코드 변경 (규칙 분리 배선) 과 런북 개정이 **다른 태스크** 였고, 태스크 리뷰는 각 diff 안에서만 정합을 본다.
- 태스크 사이의 어긋남은 whole-branch 리뷰의 **계층 간 대조** (프롬프트 ↔ enum ↔ 배선 ↔ 런북 서술) 에서만 드러난다.
- #38 (단위 테스트 · 태스크 리뷰 통과 후 계층 간 대조에서만 발견) 과 같은 패턴의 문서판 재발이다.

## 4. 예방 규칙

- **파이프라인에 분기를 추가하면 런북 스니펫을 grep 한다** — 스니펫은 로직의 사본이라 같은 커밋 (최소 같은 PR) 에서 함께 개정한다.

```bash
# 예: 분류 경로를 바꿨다면 그 함수를 인용하는 런북부터 찾는다
grep -rn "classify_stage_rows\|rows_missing_stage" docs/runbook/
```

- 런북 스니펫에는 "run.py 의 어느 블록을 미러하는지" 를 명시한다 — 미러 대상이 적혀 있어야 로직 변경 시 갱신 대상임이 드러난다.
- whole-branch 최종 리뷰 디스패치에 **계층 간 대조를 명시 항목으로 넣는다** — 이번 발견은 그 지시가 실제로 작동한 사례다.

## 5. 재발 사례 (누적 기록)

- **2026-07-20 ① 게이트 스윕 스니펫의 원문 결합 범위 불일치** (#91 최종 리뷰 블로킹 → 머지 전 정정):
  번역 게이트 런북 §5 스윕이 구단명 축을 `body_source` 단독으로 대조했는데,
  run.py 는 `title_original + body_source + body_excerpt` 결합으로 대조한다.
  좁은 원문으로 대조하면 페이월 헤드라인-온리 · fmkorea 행 (근거가 제목에만 있음) 이 전량 오탐
  → 운영자가 스윕 결과대로 재번역 큐에 넣으면 불필요한 재번역이 돈다.
  변형 교훈: 스니펫은 호출 함수만이 아니라 **대조 원문의 결합 범위까지** 1:1 이어야 한다.
- **2026-07-20 ② 사이트 재생성 예시의 서빙 계약 미반영** (#93 에서 정정):
  번역 보완 런북 §4 예시가 #87 · #88 이후 추가된 SELECT 컬럼 (published_precision · fetched_at) 과
  `outlet_dir` 인자를 반영하지 못한 상태로 남아 있었다.
  그대로 실행하면 날짜 정렬 보간 · 아웃렛 표시가 깨진 화면이 만들어진다.
  표기 백필 검증 중 발견 — 실행 전에 run.py 서빙 SELECT 와 대조해 회피했다.
- **2026-07-20 ③ 새 기본값이 기존 예시 코드의 동작을 바꿈** (#98 최종 리뷰에서 발견 → 머지 전 정정):
  차등 서빙을 넣으면서 "serving 을 지정하지 않으면 발췌 모드" 라는 기본값이 생겼다.
  그러자 서빙 검증 런북 1단계의 픽스처 예시 (serving 을 지정하지 않은 sources 설정) 가 의도와 달리 발췌 모드로 렌더돼,
  "여러 문단짜리 본문이 제대로 나오는지 확인한다" 는 픽스처의 원래 목적이 소리 없이 깨졌다.
  앞선 두 사례는 코드를 고쳤는데 예시를 같이 안 고쳐서 생긴 어긋남이지만, 이번엔 예시도 예시가 부르는 코드도 그대로였다.
  새 기본값이 "지정 안 함" 의 의미를 바꾼 것이 원인이라, 예시가 오류 없이 멀쩡히 돌기 때문에 더 찾기 어렵다.
  교훈: 새 기본값을 만들면 그 키를 안 쓰는 기존 예시 전부가 새 기본값의 영향을 받는다.
  바뀐 함수 이름으로 검색해서는 못 찾으므로 (예시가 그 함수를 직접 부르지 않음), 그 설정 (sources 등) 을 만드는 예시까지 찾아본다.
