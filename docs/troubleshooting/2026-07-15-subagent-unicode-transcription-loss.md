# Subagent 코드 전사 중 유니코드 손실 — 테스트가 못 잡는 유사 문자 강등

요약 말투 트랙 (PR #48) Task 1에서, 구현 subagent가 계획서의 정규식을 옮겨 적으며 굽은 큰따옴표 (U+201C/U+201D)를 ASCII 곧은따옴표로 바꿔 커밋한 사고.
결과 코드는 유효한 파이썬이고 테스트 7종이 전부 통과해, 태스크 리뷰의 바이트 수준 대조 전까지 아무 신호가 없었다.

## 증상

- 계획서 원본: `_QUOTED = re.compile(r'"[^"]*"|“[^”]*”|「[^」]*」|『[^』]*』|\'[^\']*\'')` — 대안 5개.
- 커밋된 코드: 두 번째 대안 (굽은 큰따옴표)이 첫 번째 (ASCII)와 동일한 중복으로 대체됨.
- 겉보기 동작 정상: 곧은따옴표 인용은 제외되므로 브리프의 인용 테스트도 통과.

## 왜 위험한가

- 이 패턴의 존재 이유가 "인용문 안 존댓말을 검출에서 빼는 것" — 한국어 뉴스 조판의 표준 인용부호가 굽은따옴표라, 손실 상태로 나가면 정상 요약을 매 회차 재생성하는 무한 백필 함정 (설계 문서가 이름 붙인 바로 그 함정)이 열린다.
- 검출을 우회하는 구조가 겹쳤다:
  - 결과가 **유효한 코드**라 lint · import · 훅 어디에도 안 걸린다.
  - 브리프의 테스트 입력도 곧은따옴표만 써서 **테스트가 손실을 공유**한다 — 같은 문서를 같은 모델이 옮겨 적으므로 테스트와 구현이 같은 방향으로 틀린다.
  - 육안 diff에서 `“`와 `"`는 거의 구분되지 않는다.

## 진단

태스크 리뷰어가 named-risk로 커밋 blob을 코드포인트 덤프해 발견했다.

```bash
git show <sha>:src/bullet_in/tone.py | uv run python -c "
import sys
for ch in sys.stdin.read():
    if ord(ch) > 127: print(hex(ord(ch)), ch)
" | sort -u
# U+201C · U+201D가 없고 ASCII 0x22만 있으면 손실
```

간편 확인은 컴파일된 패턴으로도 가능하다.

```bash
uv run python -c "from bullet_in.tone import _QUOTED; print(_QUOTED.pattern.encode('unicode_escape'))"
# 기대: “ · ” · 「~』 가 보여야 함
```

## 해결

- 패턴을 원본 5개 대안으로 복원 + **실제 U+201C/D 코드포인트가 든 회귀 테스트** 추가 (`test_curly_quoted_polite_speech_is_ignored`, 커밋 `b206844`).
- 회귀 테스트 입력은 반드시 실제 코드포인트여야 한다 — 유사 ASCII로 쓰면 테스트 자체가 손실돼 무의미해진다 (재리뷰에서 테스트 blob도 코드포인트 스캔으로 확인).

## 예방 (같은 트랙에서 실증)

- 비ASCII 리터럴 (굽은따옴표 · 이모지 · 특수 화살표)이 있는 코드를 subagent에 전사시킬 때는 **디스패치에 바이트 검증 지시를 명시**하고 검증 출력을 보고서에 첨부시킨다.
- 리뷰어 디스패치에 named-risk로 "커밋 blob의 해당 문자를 코드포인트 수준으로 확인해도 좋다"를 허용한다.
- 효과: 같은 트랙 Task 3 · 4의 프롬프트 ❌ (U+274C) · ⭕ (U+2B55)는 이 두 장치로 손실 없이 통과했고, 리뷰어가 코드포인트 재확인까지 마쳤다.
- 한계: 지시가 있어도 손실은 발생할 수 있다 (Task 1이 그 사례) — 최후 방어선은 리뷰어의 blob 대조다.
