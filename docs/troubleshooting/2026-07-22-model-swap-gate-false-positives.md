# 번역 모델을 바꾸자 제목 게이트가 오탐하기 시작함 (2026-07-22)

번역 모델을 `gemini-2.5-flash-lite` 에서 `gemini-3.1-flash-lite` 로 바꾸고 기존 224행을 전건 백필한 뒤, 영문 제목이 화면에 7건 노출됐다.
번역 품질은 오히려 좋아졌는데 (본문 말투가 어긋난 줄 11 → 0 · 파싱 실패 33% → 2.7%) 제목만 나빠진 것이 단서였다.

원인은 번역이 아니라 **게이트가 새 모델의 제목 문체를 환각으로 오인한 것**이었다.
모델을 바꿀 때마다 되풀이될 수 있어 진단 경로와 해법을 남긴다.

## 1. 증상 — 에러가 아니라 "정상 동작" 으로 보인다

- 테스트는 전부 통과하고 예외도 로그 오류도 없다.
- 게이트는 설계대로 걸렸고 폴백도 설계대로 실행됐다.
로그에도 `제목 환각 재발 — 원문 제목 폴백` 이 정상 경고로 찍힌다.
- 유일한 신호는 **폴백 행 수**다.
백필 전 1행이 백필 후 7행이 됐는데, 세어 보지 않으면 알 수 없다.

이 부류의 고장은 "무엇이 실패했는가" 를 물으면 안 잡힌다.
"무엇이 전보다 늘었는가" 를 물어야 잡힌다.

## 2. 원인 — 판정 기준이 표기 사전에 올라 있는 이름에만 묶여 있다

`enrich.detect_title_mistranslation` 은 원문 제목에 나온 인명 중 표기 사전에 올라 있는 것이 번역 제목에서 **전부** 빠졌을 때만 걸린다.
일부만 남으면 여러 소식을 담은 제목을 정당하게 줄인 것으로 보고 통과시킨다.

문제는 "일부 남았다" 를 세는 기준이 `config/name_map.yaml` 이라는 점이다.
번역 제목이 인명을 하나 살렸어도 그 이름이 사전에 없으면 살린 것으로 계산되지 않는다.

실제로 걸린 사례다.

- 원문 제목
— `Ben White is expected back in Arsenal pre-season training… Jurrien Timber is another…`
- 새 모델의 번역 제목
— `벤 화이트, 무릎 부상 회복 후 아스날 프리시즌 복귀 예정`
- 판정
— `화이트` 가 사전에 없어서 대조 대상에서 빠지고, `Timber` 누락만 남아 '전부 누락' 으로 확정

번역은 멀쩡했고 사전이 좁았을 뿐이다.
옛 모델은 제목에 인명을 여럿 나열하는 문체라 이 조건에 잘 걸리지 않았고, 새 모델이 제목을 간결하게 쓰기 시작하면서 드러났다.

## 3. 두 번째 원인 — 원문 제목이 헤드라인이 아닌 소스

트윗 소스 (`x_afcstuff`) 는 `title_original` 에 트윗 전문이 들어간다.
소스별 원문 제목 평균 길이를 재면 차이가 분명하다.

| 소스 | 원문 제목 평균 길이 |
|---|---|
| x_afcstuff | 209자 |
| football_london | 83자 |
| guardian | 77자 |
| skysports | 61자 |
| bbc_sport | 51자 |
| arsenal_official | 39자 |

즉 이 소스에서 "원문 제목의 인명" 을 세는 것은 헤드라인이 아니라 본문을 세는 셈이다.
한국어 제목은 그중 일부만 담을 수밖에 없으므로 인명 탈락이 정상이다.

라운드업 (`bbc_gossip`) 을 이 검사에서 제외한 것과 같은 이유인데, 트윗은 제외 목록에 들어 있지 않았다.
백필 폴백 7건 중 2건, 그 뒤 정기 회차의 재번역 큐 1건이 모두 이 소스였다.

실제로 걸린 트윗을 보면 이유가 분명하다.
선수 여섯 명 (스콧 · 부아디 · 바르콜라 · 알바레스 · 기마랑이스 · 콘사) 의 이적 소식을 한 트윗에 나열한 글이었고, 번역 제목은 앞머리 두 명을 담았다.
표기 사전에 있는 인명 3명이 제목에서 빠졌고, 제목에 담긴 두 명은 사전에 없어 근거가 0으로 계산됐다.

**조치 (2026-07-22 · #114)** — 이 소스에서 인명 누락 사유만 걸러낸다.

```python
BODY_AS_TITLE_SOURCES = {"x_afcstuff"}
...
if row.get("source_id") in BODY_AS_TITLE_SOURCES:
    reasons = [r for r in reasons if not r.startswith(NAME_MISSING_PREFIX)]
```

축 전체를 끄지 않은 이유는 같은 함수가 잡는 '원문에 근거 없는 임대 표현' 이 트윗에서도 유효하기 때문이다.
그 축은 원문에 `loan` 이 없으면 나올 수 없는 단어를 대조하므로 제목 길이와 무관하게 성립한다.
사유 접두어는 `NAME_MISSING_PREFIX` 상수로 두어, 검출기가 만드는 문자열과 그것을 거르는 쪽의 문자열이 어긋나지 않게 했다.

조치 후 그 트윗은 경고 없이 첫 번역에서 통과했고, 화면에도 한국어 제목이 나간다.

## 4. 진단 경로

1. **폴백 행을 센다** — `SELECT COUNT(*) FROM articles WHERE title_ko = title_original`.
교체 전 값과 비교한다.
2. **폴백 사유를 로그에서 뽑는다** — `제목 환각 재발` 경고의 `의심=` 목록.
3. **사유가 `인명 누락:` 이면 오탐을 의심한다** — 그 행의 번역 제목이 실제로 어떤 인명을 살렸는지 보고, 그 이름이 `name_map` 에 있는지 확인한다.
4. **소스를 확인한다** — 트윗처럼 원문 제목이 본문인 소스면 §3 원인이다.
5. **추측하지 말고 재번역해 본다** — 같은 행을 조건을 바꿔 가며 실제로 번역하고 게이트를 다시 통과시켜 본다.
2026-07-22 검토에서 사전 보강만은 5/7, 프롬프트 지시를 더하면 6/7 통과로 갈렸다.

## 5. 해법 — 게이트를 완화하지 않는다

검출력을 낮추면 진짜 환각까지 놓친다.
근거로 인정하는 범위만 넓히는 쪽이 맞다.

- **사전 보강** — 번역 제목이 살릴 만한 인명을 `name_map` 에 등재한다 (`화이트` · `마갈량이스` · `마르티넬리` · `비에이라` · `카비아`).
- **프롬프트 지시** — `TRANSLATE_PROMPT` 의 `title_ko` 규칙에 원문 제목의 선수 · 감독 이름을 최소 하나 남기도록 명시한다.
사전만으로는 원문 제목에 인명이 하나뿐인 경우를 손쓸 수 없다.
- **결과** — 폴백 7행이 1행으로 줄고 (남은 1건은 원문이 한국어라 폴백돼도 문제가 없다), 전체 기사의 오역 재검출이 7건에서 0건이 됐다.

사전 등재는 부작용도 함께 본다.
짧은 표기가 일반 단어에 부분 포함되면 반대 방향 오탐이 생기므로, 등재 전에 기존 키와의 부분 포함을 검사한다 (`name_map.yaml` 헤더 주의사항 ①).

## 6. 예방

- **모델을 바꿀 때마다 폴백 행 수를 전후로 비교한다** — 평가 런북의 채점 항목에 넣었다.
- **게이트 4축 재검출도 함께 잰다** — 환각 · 오역 · 라운드업 단신 누락 · 원문에 없는 구단명.
오역 축이 늘었다면 표기 사전에 빠진 이름이 있는지 먼저 의심한다.
- **원문 제목이 헤드라인이 아닌 소스를 추가할 때는 `BODY_AS_TITLE_SOURCES` 에 넣을지 판단한다.**
판단 기준은 소스별 `title_original` 평균 길이다.
기사 제목은 39~83자인데 트윗은 209자였다 (§3 표).
- **축 전체를 끄기 전에 축 안의 사유별로 나눠 본다** — 한 검출기가 여러 사유를 반환하면 그중 하나만 성립하지 않는 경우가 있다.
검출이 0건이라고 쓸모없는 것이 아니라 아직 해당 사례가 없었을 뿐일 수 있다.
