# 다중 PR 병렬 세션의 정합 함정 4종 — 스택 충돌 · 커밋 유실 · 데이터 작업 어긋남 (2026-07-19)

트랙 ③ + 상세 점검 세션에서 PR 9건 (#72~#80) 을 병렬 브랜치로 진행하며 밟은 함정의 기록.
개별 PR 은 전부 정상이었는데, **브랜치가 여러 개 열려 있는 상태 자체**가 네 가지 사고를 만들었다.
같은 규모의 병렬 세션을 다시 열 때 이 문서부터 볼 것.

## 함정 ① squash 머지 × 스택 PR — 부모가 머지될 때마다 자식이 충돌

- **증상**: 순차 머지 중 스택 자식 PR (#76 · #78 · #80) 이 전부 CONFLICTING.
- **원인**: squash 는 부모 브랜치의 원본 커밋들을 **새 커밋 하나로 대체**해 main 에 넣는다.
  git 3-way 병합은 main 의 squash 결과와 자식이 가진 부모 원본 커밋을
  "같은 위치 (anchor) 에 대한 서로 다른 삽입" 으로 취급한다
→ 내용이 같아도 자동 해소되지 않고 충돌로 판정된다.
- **오판 주의**: "내용이 동일하니 깨끗이 머지된다" 는 예측은 틀린다 (이번 세션에서 PR 본문에 그렇게 썼다가 실측 반증).
- **해소 절차** (부모 머지 직후, 자식마다 반복):
  1. `git fetch origin` 후 자식 브랜치에 `git merge origin/main`.
  2. 충돌 해소는 아래 함정 ② 절차로.
  3. `git diff origin/main --stat` 으로 diff 가 자식의 순수 변경만 남았는지 확인.
  4. push → GitHub 재계산 (수십 초 지연) 후 CLEAN 확인.
- **예방**: 스택은 꼭 필요할 때만 (같은 코드 영역 공유) 쓰고, 쓴다면 머지 라운드마다 위 재정리 비용을 계산에 넣는다.
  같은 파일 **끝에 append** 하는 병렬 PR (#75 · #79 의 테스트 파일) 도 같은 anchor 충돌을 낸다
— 병렬로 갈 파일은 삽입 지점을 분리할 것.

## 함정 ② 통짜 `checkout --ours` — main 쪽 자동 병합분 유실

- **증상**: 충돌 해소 후 main 에만 있던 변경 (#75 의 `outlet_directory` import) 이 사라짐.
- **원인**: `git checkout --ours <파일>` 은 충돌 블록만이 아니라 **파일 전체**를 우리 쪽 (stage 2) 으로 되돌린다
→ 충돌 마커 밖에서 이미 자동 병합돼 있던 main 쪽 변경까지 함께 유실된다.
- **해소**: 마커가 있는 작업 트리 파일에서 **충돌 블록만** ours 를 채택하는 방식으로 해소한다
  (블록 밖 자동 병합분은 그대로 보존됨).
  각 블록의 theirs 쪽에 고유 내용이 있는지 먼저 훑고, 있으면 수동 병합.
- **검증**: 해소 후 main 쪽 고유 심볼 (이번 사례 `outlet_directory`) 을 grep 으로 확인.

## 함정 ③ AA (양쪽 신규 추가) 파일 — `UU` grep 에 안 잡혀 마커 누출

- **증상**: 충돌 해소 커밋에 `<<<<<<<` 마커가 그대로 들어감 (`config/name_map.yaml`).
- **원인**: 양쪽이 **같은 파일을 새로 추가**한 충돌은 상태가 `UU` 가 아니라 `AA`
→ `git status -s | grep '^UU'` 로 충돌 목록을 뽑으면 누락되고, `git add -A` 가 마커째 스테이징한다.
- **탐지가 늦는 이유**: yaml 은 로드 시점에야 터지는데 테스트가 그 파일을 로드하지 않으면 통과해 버린다.
- **해소 · 예방**: 충돌 목록은 `git status -s | grep -E '^(UU|AA|AU|UA)'` 로 뽑고,
  커밋 전 `git grep '<<<<<<<'` 스캔을 습관화한다 (머지된 main 에도 한 번 더).

## 함정 ④ 머지된 PR 브랜치에 push — 커밋이 조용히 유실

- **증상**: 전부 머지 완료 후 GitHub 에 "`<브랜치>` had recent pushes" 배너
  (= 최근 push 가 있는데 연결된 열린 PR 이 없다는 신호).
- **원인**: #72 가 squash 머지된 지 38분 뒤, 상태 확인 없이 그 브랜치에 추가 커밋 (glossary 9쌍) 을 push
→ 닫힌 PR 의 브랜치라 main 에 영영 반영되지 않음.
- **선례와의 관계**: PR #34→#35 사례 (열린 PR 에 push 했으나 머지가 옛 head 로 실행돼 유실) 의 **역방향**
— 그쪽은 "머지 전 head 갱신 확인", 이쪽은 "push 전 PR 생존 확인" 이 빠진 것. 쌍으로 기억할 것.
- **회수**: 유실 커밋을 main 기준 새 브랜치로 `cherry-pick` → 회수 PR (#81). 배너 원인 브랜치는 원격 삭제.
- **예방**: 열린 PR 브랜치에 추가 push 하기 전 `gh pr view <n> --json state,mergedAt` 로 머지 여부부터 확인한다.
  사용자가 세션 중간에 머지하고 있을 수 있다 — "아까 열려 있었다" 는 근거가 아니다.

## 함정 ⑤ 파일이 안 겹쳐도 런타임에 깨진다 (2026-07-21)

- **상황**: 번역 트랙과 UI 트랙이 병렬 진행하며 파일 단위 소유권 경계를 정했다.
UI 트랙은 `serve/` 를, 번역 트랙은 `enrich.py` · `run.py` · config 를 갖는 식이다.
- **증상이 될 뻔한 것**: 경계를 지켜도 서빙이 깨질 수 있는 경로가 둘 있었다.
  - `serve/render.py` 가 `enrich.py` 의 표지 추출 함수를 import 한다
— 번역 트랙이 그 함수의 반환 형태를 바꾸면 파일은 안 겹쳤는데 서빙이 죽는다.
  - 가십 단신 카드는 **번역문** 문단 끝의 `(출처)` 표기를 원문 표지 집합과 대조해 만든다
— 번역 모델이나 프롬프트를 바꿔 그 표기가 번역되거나 위치가 옮겨지면 **git 충돌도 테스트 실패도 없이** 항목화만 사라진다.
- **원리**: 파일 소유권은 **동시 편집 충돌** 만 막는다.
모듈 경계를 넘는 import 와 한쪽이 만든 데이터를 다른 쪽이 읽는 관계는 파일 경계와 무관하게 남는다.
- **규칙**: 소유권 경계를 정할 때 파일 목록만 나열하지 말고 **경계를 넘는 import 와 데이터 의존을 함께 적는다.**
그 위에 "이 함수의 시그니처 · 반환 형태를 바꾸면 먼저 알린다" 같은 계약을 명시한다.
- **감시**: 데이터 의존은 이미 있는 게이트로 감시되는 경우가 많다.
위 사례는 `detect_roundup_omission` 이 같은 표지 집합으로 번역문을 대조하고 있어, 교체 전후 `단신누락` 로그 건수 비교로 회귀를 잡을 수 있었다 (새 가드 불필요).
다만 원문에 표지가 없는 기사는 두 축 모두 0이라 이 방법으로는 안 잡힌다.

## 함정 ⑥ 문서의 틀린 전제가 다른 세션의 판단을 바꾼다 (2026-07-21)

- **증상**: 병렬 세션이 전건 재번역을 "무료 티어 15 RPM 이라 며칠 걸린다" 고 평가해 백필을 권하지 않았다.
실제로는 유료 등급이고 십수 분 · ₩850 짜리 작업이었다.
- **원인**: `CLAUDE.md` 의 함정 항목이 "Gemini 무료 티어 … 결제 불필요" 라고 적고 있었다.
이 파일은 모든 세션이 시작할 때 읽으므로, 틀린 한 줄이 세션마다 같은 오판을 재생산한다.
- **왜 안 걸러졌나**: 그 줄은 작성 시점엔 맞았고 (무료 티어 사용 중), 이후 유료 전환이 문서에 반영되지 않았다.
코드처럼 테스트가 깨지지 않아 낡은 채로 남는다.
- **규칙**: 결제 · 한도 · 용량처럼 **외부 상태를 서술한 문장** 은 확인 시점을 함께 적고, 판단 근거로 쓰기 전에 실물을 다시 본다.
- **정정 순서**: 같은 오류가 여러 문서에 있으면 **로드 범위가 넓은 것부터** 고친다.
`CLAUDE.md` (전 세션 로드) → spec · 런북 순. spec 만 고치면 다음 세션이 같은 오판을 반복한다 (#106).

## 부록 — 브랜치 분기 상태의 데이터 작업 어긋남 (코드 함정 아님 · 운영 함정)

- **사고 2건**: ①gossip 45건 재번역을 glossary 확장이 없는 스택 브랜치에서 실행
→ 교정했던 표기 (쿄케레스) 재유입 ②site 재렌더를 이미지 제외 (#77) 미포함 브랜치에서 실행
→ 사용자 화면에 고친 문제가 되살아난 것처럼 보임.
- **원리**: DB 백필 · site 렌더는 **실행 시점 체크아웃의 코드 · config** 를 쓴다.
  브랜치가 갈라져 있으면 "어느 PR 의 수정이 반영된 실행인지" 가 매번 달라진다.
- **규칙**: 데이터 백필 · 렌더는 통합 상태에서만 실행한다
— 전부 머지된 main 이 원칙, 머지 전에 필요하면 오픈 브랜치를 전부 합친 로컬 통합 브랜치 (일회용 · 미push) 에서.
