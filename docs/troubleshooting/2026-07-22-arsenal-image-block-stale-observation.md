# arsenal.com 이미지 차단 — 낡은 관측이 코드에 굳어 히어로를 밀어냈다 (2026-07-22)

톱스토리 선정 코드가 `assets.arsenal.com` 이미지를 브라우저 차단으로 간주해 히어로 후보에서 제외하고 있었다.
그 탓에 구단 공식 (최상위 신뢰도) 오피셜 기사가 가장 큰 자리에 오르지 못했다.
재측정 결과 이미지는 차단되지 않았다 — 과거 한때 참이던 관측이 코드에 하드코딩으로 남아 계속 작동한 사고다.

## 1. 증상

- 트로사르 베식타스 방출에 `Arsenal.com` 오피셜 기사가 두 건 있는데 히어로는 BBC (공신력 상) 기사였다.
- 상위 등급 기사가 있는데도 왜 밀렸는지가 화면만 봐서는 안 보였다.

## 2. 코드에 박혀 있던 관측

목업 `build.py` 의 `_img_ok` 가 arsenal.com 을 무조건 배제하고 있었다.

```python
if (urlparse(url).hostname or "").endswith("arsenal.com"):
    return False   # "httpx 로는 200 이지만 브라우저 렌더가 차단" 이라는 주석
```

앞 UI 설계 스펙 §16.1 도 같은 내용을 별도 트랙 후보로 적어 두었다 (`ERR_BLOCKED_BY_ORB`).
히어로 선정은 "이미지가 있는 첫 기사" 를 고르므로, 배제된 오피셜은 자동 탈락했다.

## 3. 진단 — 세 층을 분리해 측정

브라우저에서 직접 재현하며 층을 갈랐다.

- **이미지 자체가 차단되는가** — localhost 페이지에서 `new Image()` 에 `referrerPolicy = "no-referrer"` 를 걸어 해당 URL 을 로드했다.
원본 크기 (1800×1200) 로 정상 수신됐다.
차단이 아니다.
- **페이지의 `<img>` 태그만 실패하는가** — 같은 URL 인데 DOM img 는 `naturalWidth = 0` 이고 `new Image()` 는 성공했다.
속성 (`referrerpolicy` · `loading`) 은 동일했다.
- **왜 DOM img 만 0 인가** — `loading = "lazy"` 였다.
측정 시점에 아직 로딩 전이었을 뿐, 시간을 두고 다시 재면 뷰포트 안 이미지 열한 개가 모두 로드됐다 (arsenal · nyt · twimg · 365dm · dailymail).

세 층의 결론 : 이미지는 차단되지 않고, 회색 플레이스홀더는 lazy 로딩 지연이었으며, 코드의 배제 규칙만 낡은 관측에 기대 있었다.

## 4. 해결

- `_img_ok` 의 arsenal.com 하드코딩 배제를 제거했다.
히어로가 구단 공식 오피셜 (트로사르) 로 바뀌고 이미지도 떴다.
- 스펙 §16.1 을 "해소됨" 으로 정정하고, 톱스토리 선정에 "이미지 유무로 신뢰도를 밀어내지 않는다" 규칙 (§5.1) 을 넣었다.

## 5. 교훈

- **외부 사이트의 동작 관측은 시간이 지나면 뒤집힌다.**
호스트 차단 · 셀렉터 · 페이월 같은 관측을 코드에 하드코딩하면, 그 사이트가 정책을 바꿔도 코드는 옛 세계를 계속 산다.
소스 셀렉터 드리프트 (`config/sources.yaml`) 와 같은 계열의 함정이다.
- **관측을 배제 규칙으로 바꿀 때는 재측정 시점을 주석에 남긴다.**
"차단됨" 만 적으면 언제 참이었는지 알 수 없어 아무도 다시 확인하지 않는다.
- **비어 보이는 이미지를 차단으로 속단하지 않는다.**
`new Image()` 로딩 · DOM img 상태 · lazy 로딩 타이밍은 서로 다른 층이다.
층을 분리해 측정해야 원인이 드러난다.
