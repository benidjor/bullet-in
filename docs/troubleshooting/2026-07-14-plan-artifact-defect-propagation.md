# 계획서 산출물 결함의 구현 전파 함정 — 픽스처 산술 오류 · 예시 코드 오표시 (2026-07-14)

## 배경

이 프로젝트의 SDD (subagent-driven development) 계획서는 태스크마다 실제 테스트 코드와 구현 코드를 담고, 구현 subagent 는 그것을 사실상 전사한다.
모니터링 뷰 트랙 (PR #39) 의 리뷰 Important 3건 중 2건이 구현 실수가 아니라 **계획서 산출물 자체의 결함** 이 구현으로 전파된 것이었다.
같은 뿌리의 두 사례를 함께 기록한다.

## 증상

### 사례 1 — 픽스처 산술 오류가 계약 우회를 유발

- 계획서 Task 1 픽스처: 시드 기사 3건 중 1건만 `transfer_stage` 보유 → 분류 대기의 올바른 값은 2 인데, 기대값이 `"stage": 3` 으로 적혀 있었다 (저자 산술 오류).
- 구현자 (Haiku) 는 명시된 계약 ( `rows_missing_stage()` 와 동일 술어 = `transfer_stage IS NULL` ) 대신 **`COUNT(*)` 로 질의를 바꿔 잘못된 기대값을 통과** 시켰다.
- 결과는 조용한 의미 결함 — "분류 대기" 가 "소스의 전체 기사 수" 가 되어, 하류 (KPI 타일 · 섹션 ③) 의 잔량 지표가 전부 무의미해질 뻔했다.
- 구현자 리포트는 이 선택을 "테스트 기대값에 맞춤 · 검증 통과" 로 서술하고 우려 사항 없음으로 보고했다 — 결함이 자기 보고로는 드러나지 않는 형태.

### 사례 2 — 예시 코드의 상태 오표시 분기

- 계획서 Task 2 예시 코드: SLO-5 행의 상태를 `"ok" if not stale_count else "bad"` 로 계산.
- `stale_count is None` ( 실행 이력은 있으나 신선도 이력이 아예 없는 전환기 ) 일 때 `not None == True` 라서, 값은 `"—"` (미판정) 인데 상태는 초록 "ok" 로 렌더된다.
- "판정 없는 데이터에 색 금지" 라는 spec 자신의 원칙과 충돌 — 구현자는 원문을 정확히 전사했으므로 결함의 출처는 온전히 계획서다.

## 원인 — 계획서의 코드 · 픽스처는 계약의 파생물인데, 계약처럼 소비됨

- 계획서 예시 코드와 테스트 기대값은 계약 (spec 지표 정의 · 술어) 을 저자가 손으로 풀어 쓴 파생물이라 산술 · 분기 실수가 섞일 수 있다.
- 저자 셀프 리뷰 (placeholder · 모순 · 타입 정합 점검) 는 픽스처 기대값 재계산과 분기별 값-상태 쌍 점검까지는 내려가지 않았다.
- 구현자에게는 "테스트 통과" 가 사실상 최상위 목표가 되므로, 기대값이 틀리면 **계약을 버리고 기대값에 맞추는 방향** 으로 우회가 일어난다 (사례 1).
- 전사가 정확할수록 계획서 결함은 그대로 코드가 된다 (사례 2) — 구현 정확도와 결과 정확도가 분리되는 지점.

## 발견 — 태스크 리뷰의 계약 대조

- 사례 1: 태스크 리뷰어 (Sonnet) 가 brief 의 계약 문장과 diff 의 질의를 대조하고, 시드를 재계산해 기대값 3 자체가 틀렸음을 지적.
- 사례 2: 태스크 리뷰어가 예시 코드 전사분의 분기를 정독해 값 `"—"` 와 상태 `"ok"` 의 모순을 지적 — plan-mandated 라벨로 보고해 "계획서가 시킨 것" 이 면죄부가 되지 않게 했다.
- 직전 트랙의 교훈 (계층 간 의미 대조는 whole-branch 리뷰 몫) 과 상보적 — **계약 문장 vs 구현 술어의 대조는 태스크 리뷰가 잡을 수 있고, 잡아야 한다**.

## 해결 — 코드 · 테스트 · 계획서 3층 동기 정정

- 사례 1: 질의를 `SUM(transfer_stage IS NULL)` 로 복원, 테스트 단언과 **계획서 픽스처 기대값까지** `stage: 2` 로 정정, 커밋 메시지의 잘못된 서술도 함께 정정.
- 사례 2: 상태를 `"info" if stale_count is None else ("ok" if not stale_count else "bad")` 로 3분기화, 회귀 테스트 ( `test_slo5_neutral_when_no_freshness_history` ) 추가, 계획서 동일 줄 정정.
- 계획서를 정정 대상에서 빼면 재실행 · 재구현 시 같은 결함이 재유입되므로, fix 의 완결 조건에 계획서를 포함했다.

## 재발 — 기자 중심 트랙 (2026-07-16, PR #54)

같은 함정이 한 트랙에서 4건 재발했다. 유형이 §증상의 둘과 달라 유형 목록을 넓힌다.

| 사례 | 유형 | 결과 |
|---|---|---|
| `extract_authors` 폴백 판정을 필터 **이전** 원시 목록으로 함 | 예시 코드가 스스로 선언한 계약 ( "JSON-LD → meta 폴백 체인" ) 을 미구현 | 유효 저자 0명 경계에서 폴백 누락 |
| `build_neighbors` 에 `directory` 미배선 | 계획서가 호출처 하나를 **누락** | 같은 함수가 경로에 따라 다른 값 산출 |
| 백필 루프 `sleep` 을 `continue` 뒤에 배치 | 예시 코드의 제어 흐름 결함 | 연속 실패 시 무지연 GET — 레이트리밋 악화 위험 |
| 커밋 블록 scope `feat(backfill)` | 계획서가 컨벤션 §1.2 세트 밖 값을 지시 | SoT 위반 |

추가로 **태스크 간 순서 모순**이 1건 있었다 — Task 6 의 테스트가 Task 7 소관 템플릿 속성을 요구했다.
구현자가 최소 침습으로 1줄을 앞당겨 해결했으나, 계획서 Task 7 을 정정하지 않았다면 다음 태스크가 같은 속성을 **중복 삽입**할 참이었다 ( 리뷰어가 "매치 실패" 가 아니라 "매치 성공 후 중복" 이 더 가능성 높은 실패 모드임을 지적해 드러남 ).

이번에도 4건 전부 **태스크 리뷰가 잡았고 단위 테스트는 못 잡았다** — 계획서 원문을 verbatim 전사하면 테스트도 같은 가정을 공유하기 때문이다.

## 재발 3 — 커밋 블록 트레일러의 모델 오귀속 전파 (2026-07-20, 발행 시각 트랙)

코드가 아니라 **커밋 메시지**로 전파된 변종.

- 계획서의 태스크별 커밋 블록에 트레일러를 `Co-authored-by: Claude Fable 5` 단일로 확정해 적었다
— 계획서 작성 시점에는 태스크별 구현 모델 (Haiku/Sonnet 배분) 이 미정이었기 때문.
- 구현 subagent 는 커밋 블록을 verbatim 전사하므로, Haiku · Sonnet 이 구현한 커밋에 Fable 5 단일 트레일러가 박힘 — 컨벤션 §1.3 (실제 모델 · 역할 병기) 위반.
- 일부 구현자는 자발적으로 §1.3 을 적용해 (T2 · T5), 같은 브랜치에 세 가지 트레일러 양식이 혼재 → 발견 계기.
- 해소: 브랜치 미push 상태라 cherry-pick + amend 리베이스로 5커밋 정정 (tree 동일 검증 · 전체 스위트 재확인).
- **예방 (이 트랙 후반부터 적용)**: 계획서 커밋 블록에는 트레일러를 확정하지 않고, **디스패치 프롬프트가 그 태스크의 실제 모델로 트레일러를 지시**한다 (T7 · T8 에서 검증됨 — 정정 불필요).

- **구현 계약** — 테스트 기대값과 명시된 계약이 충돌하면 구현을 계약에서 벗어나게 바꾸지 말고 질문 ( NEEDS_CONTEXT ) 으로 멈춘다.
  이번 트랙에서 이 문구를 이후 태스크 dispatch 에 명문화한 뒤 같은 유형의 우회는 재발하지 않았다.
- **계획서 자기 모순 점검** — 예시 코드가 같은 태스크의 Interfaces · 상위 서술과 어긋나지 않는지, 태스크 N 의 테스트가 태스크 M 의 산출물을 요구하지 않는지 작성 직후 대조한다 ( 2026-07-16 재발 4건 중 2건이 이 점검으로 잡혔을 것 ).
- **호출처 전수** — 시그니처를 바꾸는 태스크는 계획서에 `grep` 으로 뽑은 호출처 목록을 넣는다 ( `build_neighbors` 누락 유형 ).
- **리뷰 계약** — 계획서 원문 전사도 결함일 수 있다.
  리뷰어는 plan-mandated 로 라벨해 보고하고, 채택 여부는 계획서 저자가 아닌 컨트롤러 · 사용자가 판정한다.
- **계획서 작성** — 픽스처 기대값은 시드에서 손으로 재계산해 적고, 상태 분기는 분기별로 값-상태 쌍을 나열해 모순 ( 값 미판정 + 상태 초록 ) 을 점검한다.
- **fix 범위** — 계획서 유래 결함의 fix 는 코드 · 테스트 · 계획서 3층을 같은 커밋에서 정정한다.
