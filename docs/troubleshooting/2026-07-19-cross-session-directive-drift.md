# 세션 간 지시 표류 — 기록된 원 지시와 후속 발언 충돌

- 날짜: 2026-07-19
- 관련: 트랙 ② (PR #60) football.london 처리 · spec `docs/superpowers/specs/2026-07-19-source-curation-design.md` §1

## 1. 증상

같은 작업 항목에 대해 세션마다 다른 처리 방침이 기록에 남는다.

- 원 지시 (2026-07-16): football.london 은 "Tom Canton 만 허용 · 나머지 DB 삭제 · 신규 필터 차단".
- 후속 세션 (2026-07-19, 트랙 ①): 같은 항목을 "서빙 숨기기 · 트랙 ③" 으로 지칭 — 처리 방식과 트랙 번호가 모두 원 지시와 다름.

## 2. 왜 위험한가

- 새 세션은 최근 기록을 우선 신뢰하기 쉬워, 충돌을 인지하지 못하면 **임의로 한쪽을 골라 진행**한다.
- 이 사례는 비가역 작업 (기사 157건 삭제) 이 걸린 결정 — 잘못 고르면 데이터 복구 불가 또는 의도 왜곡.
- 표류는 대화 중 지나가는 재조정 발언에서 생기기 쉽다
— 발언 시점엔 사용자도 원 지시와의 충돌을 인지하지 못할 수 있다.

## 3. 진단

- 결정 이력 대조: 메모리 · spec · PR 본문에서 같은 항목의 서술을 시점순으로 모아 비교한다.
- 이 사례의 실제 발견 경로: 트랙 ① 세션이 spec 비목표에 "서빙 숨기기" 로 적다가 원 지시와 다름을 자각
→ 세션 종료 스냅샷에 ⚠️ 로 명시 기록.

## 4. 해결

- 충돌을 **착수 게이트로 승격**: 다음 세션 시작 프롬프트에 "삭제 vs 숨기기를 물어보고 시작할 것" 을 명시.
- 트랙 ② 착수 시 선택지 3안 (삭제 + 차단 / 차단 + 숨김 절충 / 숨기기만) 을 장단점과 함께 제시
→ 사용자가 원 지시 (삭제 + 차단) 재확정 → spec §1 에 재확정 경위를 기록.

## 5. 예방

- 충돌을 인지한 세션이 **침묵하지 않고 ⚠️ 로 즉시 기록**한다 (이번 사례의 방어선).
- 비가역 작업이 걸린 충돌은 메모리 기록만으로 부족
— 다음 세션이 무조건 지나가는 **프롬프트 레벨 게이트** ("물어보고 시작") 로 올린다.
- 결정을 번복할 때는 번복 사유와 함께 SoT 문서에 남긴다
— 적용례: bbc_gossip "썸네일만" → "본문 전체 번역" 번복을 사유와 함께 백로그 SoT (`docs/superpowers/2026-07-19-post-v1-followup-tracks.md` §0 · §4) 에 기록.
