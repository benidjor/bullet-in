# 서버 이전 계획에서 DB 데이터 이관이 빠지는 함정 — 첫 회차가 성공해 유실을 가림

- 날짜: 2026-07-20 (스케줄 상시 가동 트랙의 VM 이전 작업 중 발견 · 세션 내 해소)
- 같은 부류: 문제가 있는데 겉보기엔 정상인 유형 (`2026-07-19-silent-zero-collection-blindspot.md` 의 수집 0건 사각과 같은 계열)
- 조치: 로컬 DB 를 VM 으로 이관 (plan Task 4.5 로 소급 기록) · 절차는 VM 동거 런북 §4

## 1. 증상

- VM 을 처음 세팅하고 돌린 첫 회차가 **67건 수집 · 오류 없음 · 성공률 1.0** 으로 깨끗하게 끝났다.
- 그런데 이 67건은 텅 빈 DB 위에 새로 수집된 것이었다.
  로컬에 쌓여 있던 기존 205건과, 그 위에 손으로 고쳐 온 이력 (발행 시각 보정 · 인명 표기 정정 · 번역 환각 정정) 이 VM 에는 하나도 없었다.
- 이 상태로 스케줄을 켰다면 공개될 사이트는 67건짜리가 되고, 과거 기사와 수작업 정정을 전부 잃은 채 운영이 시작됐을 것이다.

## 2. 원인 — 이전 계획이 실행 환경만 다루고 데이터를 빠뜨림

- 이전 계획 (spec §2.1 · plan) 에 적힌 옮길 것은 코드 · 라이브러리 · 비밀 값 · 스케줄 · 컨테이너, 즉 **실행 환경** 뿐이었다.
- DB 볼륨 · 회차 이력 테이블 같은 **데이터** 는 어디에도 명시되지 않았다.
  "현 구성을 그대로 옮긴다" 는 문구가 데이터까지 포함하는 것처럼 읽히지만, compose 를 새 서버에서 올리면 볼륨은 빈 채로 새로 만들어진다.

## 3. 왜 어떤 지표도 잡지 못하나

- 회차 지표 (수집 건수 · 오류 · 성공률) 는 전부 정상이다.
  빈 DB 는 회차 입장에서 "신규가 많은 정상 상태" 와 구분되지 않는다.
- 신선도 · 수집량 이상 알림도 울리지 않는다 — 비교 기준선 자체가 새 DB 에서 다시 시작되기 때문이다.
- 이번에 잡아낸 방법은 지표가 아니라 **사람의 기대치 대조** 였다.
  회차 결과 (67건) 가 기존 DB 건수 (205건) 와 안 맞는다는 것을 알아챈 것이 유일한 검출 경로였다.
→ "첫 회차가 성공했다" 는 것은 이전이 끝났다는 증거가 되지 못한다.

## 4. 해소

- 로컬에서 덤프 (mariadb-dump · mongodump --gzip) → scp 전송 → VM 에서 복원 (기존 테이블 대체 · mongorestore --drop).
- 검증은 건수 대조로: articles 205 · pipeline_runs 19 · mongo raw 442 가 로컬과 일치 + 사이트 재생성 205 페이지.
- 복원 때문에 이관 전 VM 회차가 새로 수집한 2건은 사라졌지만, 소스 페이지에 아직 떠 있는 글이라 다음 회차가 다시 수집한다 (손실을 수용한 근거).
- 상세 절차: `docs/runbook/2026-07-20-vm-cohost-bootstrap.md` §4.

## 5. 예방 규칙

- 서버 이전 · 이주 계획에는 옮길 것 다섯 가지를 빠짐없이 적는다:
  **① 코드 ② 의존성 (파이썬 패키지 · 브라우저) ③ 비밀 값 (.env · 쿠키) ④ 스케줄 ⑤ 데이터 (DB 볼륨 · 이력 테이블 · 생성된 사이트 파일)**.
  ⑤ 가 빠진 계획은 "그대로 옮긴다" 고 써 있어도 데이터를 옮기지 않는다.
- 이전이 끝났는지는 "첫 회차 성공" 이 아니라 **옮기기 전후의 DB 건수 대조** (기사 · 회차 이력 · 원문 저장소) 로 판정한다.
- 앞으로 다시 이주할 일이 생기면 (유료 VM 전환 · Oracle 정책 변동) 같은 런북 §4 를 그대로 쓴다.
