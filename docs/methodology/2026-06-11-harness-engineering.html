1

이 범주는 "하네스 엔지니어링"이 맞습니다

코드를 직접 쓰는 게 아니라, LLM이 작동하는 발판(프로세스·컨텍스트·검증 루프)을 설계하는 일입니다. Karpathy의 표현으로는 "명령형 → 선언형(목표 + 검증 루프) 전환". 다만 한 덩어리가 아니라 결이 다른 4종이 섞여 있습니다.

프로세스 하네스
작업 생애주기 파이프라인 — superpowers, ouroboros
행동 가드레일
코딩 중 지켜야 할 원칙 — karpathy-guidelines
오케스트레이션 하네스
대규모 병렬 에이전트 — dynamic workflows
계획·리뷰 표면
계획 검토 UX — ultraplan
2

본질 + superpowers와의 중첩

핵심은 중복 제거입니다. 중첩이 클수록 추가 가치는 작고 의식(ceremony)만 늘어납니다.

도구한 줄 정체중첩도고유 가치
superpowers (현재) brainstorm → plan → 서브에이전트 구현 + 2단계 리뷰 → finish 기준 풀 생애주기 + TDD + 리뷰 게이트
karpathy 4대 행동지침(가정점검·단순화·수술적변경·목표주도) 낮음 ✅ 오버헤드 0, 코딩 중 품질 가드레일
ouroboros spec-first 런타임(Interview→Seed→Execute→Evaluate→Evolve) 매우 높음 ⚠️ 모호도<0.2 게이트, 멀티모델 합의 검증
dynamic workflows 수십~수백 에이전트 병렬 오케스트레이션 보완적 대규모 폭(감사·마이그레이션), 적대적 검증
ultraplan 계획을 클라우드로 보내 인라인 리뷰 후 실행 보완적 큰 계획의 풍부한 리뷰 UX (GitHub+웹 필요)
판정 · ouroboros는 superpowers와 거의 1:1 중복 → 전체 도입 대신 아이디어 2개만 차용. dynamic workflows는 "폭"용 → 깊이-1 작은 기능엔 과함. ultraplan은 방법론이 아니라 리뷰 장소 차이 → 큰 계획에만. karpathy만 중첩이 작고 오버헤드 0 → 즉시 채택.

권고 — "린 스택"

지금 작업의 최적 조합
superpowers+karpathy
척추(superpowers) + 얇은 가드레일(karpathy). 나머지 3종은 예비로 두고 트리거될 때만 꺼냅니다.
우선 ①④ 효율성·속도
겹치는 ouroboros 전체·ultraplan·workflows를 빼서 의식을 최소화. 네 개 다 얹는 게 가장 느립니다.
우선 ② 구현 용이성
karpathy는 서브에이전트 프롬프트에 4원칙 한 단락 주입만으로 끝. 설치·학습 0.
우선 ③ 정확성
"가정 점검 + 수술적 변경"이 LLM 대표 실수(엉뚱한 수정·과설계)를 직접 차단. 2단계 리뷰가 보강.

6

지금 작업에 당장 반영 — karpathy 4원칙 주입

멈춘 지점(T1 구현)을 다음만 바꿔 재개합니다. 각 implementer / reviewer 서브에이전트 프롬프트 상단에 주입:

1
가정 표면화
코딩 전 가정·트레이드오프를 드러내고, 불명확하면 질문 (Think Before Coding)
2
단순함 우선
요청된 것만, 투기적 추상화 금지 (Simplicity First · YAGNI)
3
수술적 변경
무관한 코드·주석은 건드리지 않기 (Surgical Changes)
4
목표 주도
검증 가능한 성공기준(테스트 통과)으로 스스로 루프 (Goal-Driven)
superpowers 흐름(brainstorm→plan→subagent구현+2단계리뷰)은 그대로 유지, karpathy를 얇게 덧입히는 방식. T1부터 이 오버레이로 재개하면 됩니다.
